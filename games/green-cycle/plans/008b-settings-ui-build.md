## Why this matters

Plan-008 wired `SaveData.settings.volume` to the audio engine, but the player has **no way to change it from inside the game**. The audio gain is loaded from the saved default (`1`) and stays there. The `quality: 'low' | 'high'` field is also dead weight in `SaveData`: it is persisted, loaded, and never read. This plan surfaces both fields on the start-menu overlay and implements the `quality: 'low'` runtime branch so the field earns its keep.

## Goal

1. Add a volume slider and a quality selector to `#overlay` (the start menu).
2. Persist user changes to `saveData.settings` and write to `localStorage` via `SaveManager.save`.
3. Apply volume changes live via `audio.applySettings(...)`.
4. Implement the `quality: 'low'` branch in `Renderer` so it is no longer dead weight.

Then, after the existing event bindings but before `syncMenu()` ([src/main.ts:142](file:///Users/doer/dev/yoo/games/green-cycle/src/main.ts#L142)), add:

```typescript
// 初始化设置控件的当前值
const initialSave = game.getSaveData();
ui.volumeSlider.value = String(initialSave.settings.volume);
ui.volumeValue.textContent = `${Math.round(initialSave.settings.volume * 100)}%`;
ui.qualitySelect.value = initialSave.settings.quality;

// 音量滑动 → 实时生效 + 持久化
ui.volumeSlider.addEventListener('input', () => {
  const v = Number(ui.volumeSlider.value);
  game.getSaveData().settings.volume = v;
  audio.applySettings({ volume: v });
  ui.volumeValue.textContent = `${Math.round(v * 100)}%`;
  SaveManager.save(game.getSaveData());
});

// 画质切换 → 持久化（运行时由 Renderer 直接读取，无需重新进入菜单）
ui.qualitySelect.addEventListener('change', () => {
  const q = ui.qualitySelect.value as 'low' | 'high';
  game.getSaveData().settings.quality = q;
  SaveManager.save(game.getSaveData());
});
```

Notes for the executor:
- The volume handler uses `input` (not `change`) so the audio gain updates while the user drags. Persistence runs on every `input` — localStorage writes are fast, but if 008c work later wants to debounce, that is a separate plan.
- The `muted` field of `applySettings` is left at its default (current `Audio.muted`) by passing only `{ volume }`. That is correct: today's `SaveData` does not have a `muted` field; plan-008 designed `applySettings` precisely for this asymmetry.
- `SaveManager` is already imported in `Game.ts` but not in `main.ts`. Add the import: `import { SaveManager } from './utils/SaveManager';`.

### Step 3: Make `Renderer` honor the quality setting

Verified: `state.saveData` does **not** exist today. `GameState` ([src/game/State.ts:36-115](file:///Users/doer/dev/yoo/games/green-cycle/src/game/State.ts#L36-L115)) has no reference to `SaveData`. The cleanest fix is to add **one optional field** to `GameState`:

```typescript
// src/game/State.ts (add to the class body near the top)
saveData: SaveData | null = null;
```

Then in [src/game/Game.ts:110-137](file:///Users/doer/dev/yoo/games/green-cycle/src/game/Game.ts#L110-L137) (constructor) and again at [src/game/Game.ts:152-174](file:///Users/doer/dev/yoo/games/green-cycle/src/game/Game.ts#L152-L174) (`startGame` — when a new `GameState` is constructed), assign:

```typescript
this.state.saveData = this.saveData;
```

The field is set in two places because `startGame` does `this.state = new GameState(path)` and the new instance starts with `saveData = null`. The double-assignment keeps the reference consistent for the lifetime of the `Game`.

Now wire the quality branch in [src/render/Renderer.ts:46-63](file:///Users/doer/dev/yoo/games/green-cycle/src/render/Renderer.ts#L46-L63):

```typescript
render(state: GameState, mouseWorld: Vec2): void {
  const ctx = this.ctx;
  const isHighQuality = state.saveData?.settings.quality !== 'low';
  ctx.save();
  ctx.fillStyle = '#000000';
  ctx.fillRect(0, 0, this.canvas.width, this.canvas.height);
  ctx.translate(this.offsetX, this.offsetY);
  ctx.scale(this.scale, this.scale);
  drawMap(ctx, state, mouseWorld);
  drawEntities(ctx, state, isHighQuality);
  drawBuildPreview(ctx, state, mouseWorld);
  drawSelectBox(ctx, state);
  if (isHighQuality) drawBossBar(ctx, state);
  if (isHighQuality) drawAlert(ctx, state);
  ctx.restore();
}
```

Then open `src/render/EntityRenderer.ts` and find `drawEntities(ctx, state)`. Change its signature to `drawEntities(ctx, state, isHighQuality: boolean)`. Inside, when spawning the per-frame "particles" for a damage flash / hit / death effect, read the flag — for `isHighQuality === false`, use the lower-cap particle counts (see step 4).

Notes for the executor:
- The optional chaining (`state.saveData?.settings.quality`) is defensive: the field is null briefly during construction, and rendering during that window would be a bug, but the safe default (high quality) is correct.
- This is a **data reference, not a copy**: `Game.saveData` is the same object held in the menu UI. Volume/quality changes from the menu propagate to the next frame's render with no extra plumbing.
- `import type { SaveData }` already exists in `State.ts` transitively (or add it to the import block at the top — it lives in `src/types.ts`).

**Verify**: `npm run typecheck` exits 0; `grep -n "isHighQuality\|settings.quality" src/render/Renderer.ts src/render/EntityRenderer.ts src/game/State.ts src/game/Game.ts` shows the new flag plumbing across all four files.

## Maintenance notes

- `Audio.applySettings` is the only aggregator between `SaveData` and the audio runtime. When a `muted` field is added to `SaveData` (a likely follow-up), update both `SaveData` and the `applySettings` call sites in `main.ts` (step 2). Plan-008 already designed the helper to accept `muted` as optional precisely for this.
- The `quality: 'low'` branch is currently a thin visual cut. If a future plan introduces more expensive visuals (e.g. screen shake, post-processing), revisit the list in step 3 and add them to the `if (isHighQuality)` gates.
- The module-level `currentQuality` flag in `EntityRenderer` works only because rendering is single-threaded and synchronous. If the render loop is ever refactored to be async or worker-based, replace the flag with an explicit parameter on each `drawX` call.
- Deferred follow-ups (out of this plan):
  - **M-key keyboard binding for mute** — small enough to be a one-line addition in `InputManager` plus a `audio.setMuted(...)` call. Schedule as plan-008c when desired.
  - **In-game settings shortcut** — a gear icon in the topbar that re-shows the settings row during a run. The current overlay-only model forces the player to lose or win before changing settings. Schedule as plan-008d if playtest demands it.
  - **Save migration** — `SAVE_VERSION = 1` ([SaveManager.ts:6](file:///Users/doer/dev/yoo/games/green-cycle/src/utils/SaveManager.ts#L6)) is unchanged by this plan; no migration needed. If plan-008c adds a `muted` field, bump to `2` and add a migration path in `load()`.
